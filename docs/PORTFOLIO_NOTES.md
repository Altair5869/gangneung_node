## 2026-07-16 — AI 큐레이터 동선 엔진 재설계

**배경**: 워크-라이프 동선 추천 기능이 "카페 여러 곳을 옮겨다니라"는 부자연스러운 결과를 내고 있음을 실사용 중 발견, 워크스팟 1곳+관광지+식당 구조로 전면 재설계.

- **LLM 출력 제약을 프롬프트가 아니라 스키마로 강제**: "워크스팟 정확히 1곳"을 자연어로 지시했지만 Claude(Haiku)가 종종 2곳을 선택하는 걸 실측 테스트로 확인. Tool use 스키마를 단일 `spotIds` 배열에서 `workSpotId`(string 1개)+`lifeSpotIds`(배열)로 분리해 애초에 "워크스팟 2곳"을 표현할 수 없는 구조로 바꿔 100% 해결. *(포인트: LLM 안정성 문제를 프롬프트 엔지니어링이 아니라 인터페이스/스키마 설계로 해결한 사례)*

- **지리 정보 부재로 인한 동선 거리 폭증 진단 및 해결**: 사용자가 "동선이 너무 멀다"고 보고 → 재현 테스트로 원인이 "관광지/식당 후보가 워크스팟 위치와 무관하게 API 응답 순서 그대로 노출됨"임을 특정. 워크스팟 후보 좌표 기준 하버사인 최단거리로 재정렬하는 필터를 추가해, 동일 시나리오 반복 테스트에서 **평균 이동거리 16~21.5km → 1.6~5.5km로 개선** (약 75~90% 감소). *(포인트: 정량 측정 가능한 개선, 같은 입력에 대해 5회씩 재테스트해서 효과를 검증)*

- **외부 API 데이터 모델링 한계로 인한 데이터 중복 버그**: 관광공사 OpenAPI가 카페와 일반 식당을 같은 `contentTypeId`(39, 음식점)로 제공한다는 사실을 모른 채 워크스팟/식당 두 파이프라인을 독립적으로 만들었다가, 같은 장소(동일 `contentid`)가 "방금 일한 카페"이자 "추천 식당"으로 동시에 노출되는 버그 발생. `id` 문자열에 인코딩된 `contentid`를 비교해 원인을 특정하고, 식당 후보 목록에서 카페류 키워드를 필터링해 해결. *(포인트: 외부 API의 암묵적 데이터 모델 가정이 깨졌을 때 생기는 버그 클래스, id 분석을 통한 근본원인 추적)*

- **생성-랭킹 파이프라인 분리로 인한 조건 무시 버그**: 자유 텍스트 검색(Voyage 임베딩)이 후보 "순서"만 바꿀 뿐, 정작 최종 선택을 하는 Claude에게는 사용자가 그 조건을 요청했다는 사실 자체가 프롬프트에 전달되지 않아 조건이 사실상 무시됨. 사용자가 "이 카페는 실제로 바다가 안 보인다"는 현지 지식으로 지적하면서 발견. 프롬프트에 조건을 명시하고, 근거 데이터(장소 설명文)를 함께 제공해 "근거 없는 감각적 묘사 생성(환각)"까지 같이 억제. *(포인트: RAG/랭킹-생성 분리 아키텍처에서 흔히 놓치는 "랭킹 신호가 생성 단계에 실제로 전달되는지" 검증의 중요성)*

- **구조화 필드 간 일관성 문제 반복 발견**: LLM이 방문 순서(구조화 배열)와 방문 순서를 설명하는 자연어 문장을 각각 독립적으로 생성해 서로 어긋나는 문제가 스키마를 리팩터링할 때마다 다른 형태로 재발(7/14, 7/16 두 번). 최종적으로 자유문장 `description`을 없애고, 순서 배열과 정확히 같은 인덱스로 짝지어지는 `stopNotes` 배열로 바꿔 "구조적으로 불일치가 불가능한" 설계로 해결. *(포인트: 같은 클래스의 버그가 인터페이스를 바꿀 때마다 재발한다는 걸 인지하고, 프롬프트 강화가 아니라 데이터 구조 자체를 병렬화해서 근본적으로 막은 사례)*

- **외부 라우팅 API 연동**: 지도에 표시되는 동선이 직선(haversine)이라 실제 도로 거리를 반영 못 하는 문제를 발견 → 카카오모빌리티 길찾기 API가 이미 프로젝트 REST 키로 활성화되어 있음을 확인하고 신규 연동, `waypoints` 파라미터로 다중 경유지를 단일 API 호출로 처리해 실제 도로 기반 폴리라인 렌더링.

- **QA 방법론**: `npx tsc`/`eslint`/`next build` 같은 정적 검증과 API curl 테스트를 통과해도 실제로 위 버그들은 전혀 안 잡혔고, 브라우저에서 실사용 스크린샷을 받은 후에야 3라운드 연속으로 새 버그가 드러남. 이후 각 라운드마다 "재현 → 원인 특정 → 수정 → 동일 시나리오 N회 재실행으로 정량 검증" 루프를 반복하는 방식으로 작업.
