# AI 에이전트 설계

`curateRoute`(`lib/ai.ts`)를 단일 프롬프트 호출에서 2노드 LangGraph 워크플로우로 바꾸는 설계. 4노드가 아니라 2노드로 좁힌 이유: 스팟 규모가 30~50곳 수준이라 노드를 늘리면 레이턴시만 늘고 검증 기준이 불명확해짐.

## 왜 필요한가 (LLM 호출을 늘리는 게 아니라 코드 검증을 추가하는 이유)

지금 `curateRoute`는 `preferences`(무장애, 콘센트 등)를 지켰는지 확인하는 코드가 없다. Claude가 조건에 안 맞는 스팟을 골라도 그대로 사용자에게 나간다. 또한 `nearestNeighborSort`는 `Math.hypot(lat 차이, lng 차이)`로 거리를 계산하는데, 이건 실제 km가 아니라 위경도 차이를 그냥 유클리드 거리로 계산한 값이다. 이 두 결함을 코드로 검증하는 게 목적이다.

## 노드 구조

**Node 1 — 생성**
지금의 `curateRoute` 로직 (RAG 검색 붙인 버전). 입력: `CurationRequest` + 검색된 후보 스팟. 출력: `CurationRoute`.

**Node 2 — 검증 (코드 기반, LLM 재호출 아님)**
아래 조건을 코드로 체크한다. 하나라도 실패하면 실패 이유를 프롬프트에 追가해서 Node 1을 재호출한다.

## preferences ↔ 필드 매핑

`PREFERENCE_OPTIONS`(`app/ai-curator/page.tsx`) 6개 중 검증 가능한 4개만 검증 대상으로 한다.

| preference 값 | 검증 여부 | 조건 | 비고 |
|---|---|---|---|
| "조용한 환경" | 검증함 (신호 기반) | `noise === "언급됨-조용함"` | 확정된 사실이 아니라 네이버 블로그 커뮤니티 신호. `noise === "언급없음"`인 스팟은 이 조건을 건너뜀 (통과도 실패도 아님). UI에 "커뮤니티 후기 기준"이라고 명시 |
| "빠른 WiFi" | **검증 안 함, 조건 자체 삭제** | — | wifi 속도(Mbps)는 전화로도 블로그로도 확인 불가능. 이 조건을 만들면 안 되는 데이터에 임계값을 거는 것이 됨 |
| "콘센트 필수" | 검증함 (사실 기반) | `power.available === true` | 전화 확인/직접 조사로 채운 값. `null`인 스팟은 건너뜀 |
| "무장애 접근 가능" | 검증함 (사실 기반) | `barrierFree !== undefined` | 실데이터이므로 엄격하게 검증 |
| "뷰 좋은 곳" | 검증 안 함 | — | 구조화 필드 없음, `tags` 매칭은 스팟마다 태그가 일관되지 않아 랜덤 통과/실패가 됨 |
| "카페인 충전 가능" | 검증 안 함 | — | 위와 동일 |

**검증 조건 두 종류를 구분한다.** "콘센트 필수", "무장애 접근 가능"은 사실 기반(fact-based) — 전화로 확인되는 예/아니오다. "조용한 환경"은 신호 기반(signal-based) — 커뮤니티 후기에 그런 언급이 있었는지일 뿐, 그 시각 그 장소가 실제로 조용한지 보장 안 한다. 이 둘을 UI에서 같은 방식으로("무장애 접근 가능" 배지처럼) 표시하면 안 된다.

**미확정 항목**: `wifi.speedMbps` 임계값 논의는 폐기함. 이제 논의할 필요 없음.

## 거리 검증

1. `nearestNeighborSort`가 쓰는 `Math.hypot` 계산을 하버사인 공식으로 교체한다. 실제 km 단위 거리 함수가 먼저 있어야 함.
2. `duration`(2/4/6/8시간)별 총 이동거리 임계값을 둔다.

| duration | 총 이동거리 임계값 (제안, 미확정) |
|---|---|
| 2시간 | 3km |
| 4시간 | 6km |
| 6시간 | 10km |
| 8시간 | 15km |

**미확정**: 이 숫자는 강릉 시내 기준 추정치이고 실측 검증 안 됨. 순차 이동거리 합으로 계산할지, 전체 스팟을 감싸는 반경으로 계산할지도 확정 안 됨 (순차 합이 구현은 쉽지만 왔다갔다하는 동선도 통과시킬 수 있음).

## 검증 대상에서 제외

`RouteStop`이 `LifeSpot`(`isLifeSpot(s) === true`)이면 wifi/power/noise/barrierFree 검증을 건너뛴다. `LifeSpot` 타입에는 이 필드들이 없다.

## 재시도 정책

- 최대 재시도 횟수: 2회 (제안, 미확정 — 무한 루프 방지 + Claude API 비용 고려)
- 재시도 시 실패 이유(예: "무장애 조건 위반: 스팟 X")를 다음 프롬프트에 명시적으로 넣는다. 이유 없이 같은 프롬프트로 재호출하면 같은 결과가 나올 확률이 높다.
- 2회 다 실패하면: 에러를 던지지 않고 "일부 조건을 만족하는 동선을 찾지 못해 근접한 결과를 보여드립니다"라는 안내와 함께 마지막 결과를 반환한다 (제안, 미확정).

## wifi 유무 확인 + noise 신호 확보 — 네이버 검색 API 에이전트

**목적**: 30곳 대상으로 wifi 유무와 noise(소음) 커뮤니티 신호를 확보. `wifi.available`, `power.available`은 전화 확인으로 채우고, `noise`는 처음부터 전화로 검증이 불가능하므로 이 방식으로만 채운다.

**금지 사항**: 네이버 지도/카카오맵 리뷰 페이지 직접 크롤링. 이용약관 위반 소지.

**허용 방식**: 네이버 검색 오픈API(블로그, `https://openapi.naver.com/v1/search/blog`). 발췌문(약 200자)만 제공되므로 이진 판단(있음/없음) 대신 3단계 라벨을 쓴다. wifi/noise 둘 다 같은 방식을 쓴다.

| 필드 | 라벨 | 조건 |
|---|---|---|
| wifi 커뮤니티 신호 | `언급됨-있음` | 발췌문에 "와이파이 있어요", "wifi 빠름" 등 긍정 표현 매칭 |
| wifi 커뮤니티 신호 | `언급됨-없음` | 발췌문에 "와이파이 안 됨", "wifi 없음" 등 부정 표현 매칭 |
| wifi 커뮤니티 신호 | `언급없음` | 매칭 안 됨 |
| noise | `언급됨-조용함` | 발췌문에 "조용해요", "작업하기 좋아요" 등 표현 매칭 |
| noise | `언급됨-시끄러움` | 발췌문에 "시끄러워요", "음악 소리 큼" 등 표현 매칭 |
| noise | `언급없음` | 매칭 안 됨 |

**wifi 필드 구조**: `wifi.available`(전화 확인, 사실 기반)과 `wifi.communitySignal`(네이버 API, 신호 기반)을 분리해서 저장한다. 둘 다 있는 경우 `wifi.available`(사실 기반)을 우선 신뢰하고 UI에 표시한다. `wifi.communitySignal`은 `wifi.available`이 `null`일 때 참고용으로만 쓴다.

**noise 필드 구조**: `noise` 필드 자체가 `"언급됨-조용함" | "언급됨-시끄러움" | "언급없음"`이다. 사실 기반 대응 필드가 없다 — noise는 애초에 전화로 확인할 방법이 없으므로 신호가 유일한 데이터 소스다.

**구현**:
- Tool: 네이버 검색 API 호출 함수 (Python), 스팟당 쿼리 2~3개 (예: "{장소명} 와이파이", "{장소명} 조용")
- LLM 역할: 발췌문 여러 개를 받아서 wifi/noise 각각 3단계 라벨 중 하나로 분류 + 근거 문장 반환 (구조화 출력)
- 적용 범위: 30곳만 (카페 21 + 호텔 6, 도서관 3은 대상에서 제외 — 도서관은 noise가 사실상 `언급됨-조용함`으로 고정되는 카테고리라 별도 확인 불필요)

**대안**: 발췌문 정확도가 낮다고 판단되면, wifi는 직접 조사(전화 문의, 카페 홈페이지 확인)로 채운다. noise는 직접 조사 대안이 없다 — 소음은 애초에 사실 기반 확인이 불가능한 속성이므로, 네이버 API 방식을 안 쓰면 `noise`는 전부 `언급없음`으로 남는다.

## 미확정 항목 정리 (구현 전 확정 필요)

1. duration별 거리 임계값 4개 숫자
2. 거리 계산: 순차 합 vs 반경
3. 재시도 실패 시 동작: fallback 문구 vs 에러
4. wifi 커뮤니티 신호(네이버 API) vs 직접 조사만 사용 최종 선택
